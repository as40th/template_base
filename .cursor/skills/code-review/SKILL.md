# Процесс Review

## 1. Определить контекст изменений

Понять:

* что изменилось
* зачем изменилось
* какой сценарий затронут

Если цель изменений неясна:

* отметить это как риск

---

## 2. Определить затронутые компоненты

Выявить:

* изменённые модули
* изменённые контракты
* новые зависимости
* новые интеграции

Проверить влияние на:

* публичные интерфейсы
* интеграции
* backward compatibility

---

## 3. Проверить соответствие структуре проекта

Сверить изменения с:

* docs/structure.md

Проверить:

* корректность размещения модулей
* соблюдение границ слоёв
* корректность зависимостей между слоями

---

## 4. Проверить соответствие ADR

Для каждого затронутого сценария определить:

* какие ADR применимы

Проверить:

* соответствует ли реализация принятому решению
* не нарушаются ли архитектурные ограничения

Все нарушения сопровождать ссылкой на ADR.

---

## 5. Проверить соответствие Data Flow

Если изменение затрагивает:

* RAG
* Tool Calling
* MCP
* Agent Flow
* Интеграции

Сверить реализацию с соответствующим data-flow.

Проверить:

* последовательность вызовов
* направление зависимостей
* корректность маршрутизации

Все отклонения сопровождать ссылкой на конкретный data-flow.

---

## 6. Проверить корректность реализации

Проверить:

* обработку ошибок
* edge cases
* пустые значения
* конкурентный доступ
* работу с состоянием

Особое внимание:

* async код
* сетевые вызовы
* операции записи

---

## 7. Проверить эксплуатационные аспекты

Проверить наличие:

* timeout
* retry
* observability
* логирования
* обработки отказов зависимостей

---

## 8. Проверить безопасность

Проверить:

* работу с секретами
* обработку пользовательского ввода
* использование внешних данных
* потенциальные утечки информации

---

## 9. Проверить поддерживаемость

Оценить:

* читаемость
* сложность
* связанность
* дублирование

Выявить:

* избыточную сложность
* скрытую бизнес-логику
* неочевидные зависимости

---

# Формат результата

## Summary

Краткая оценка изменений.

---

## Findings

Для каждого замечания:

### Severity

* Critical
* High
* Medium
* Low

### Reference

Ссылка на:

* ADR
* Structure
* Data Flow
* Rule

### Location

Файл или модуль.

### Problem

Описание проблемы.

### Risk

Почему это опасно.

### Recommendation

Рекомендуемое исправление.

---

## Positive Findings

Перечислить удачные решения:

* соблюдение ADR
* корректное использование паттернов
* улучшение архитектуры
* снижение связанности
* улучшение наблюдаемости

---

## Final Verdict

Один из вариантов:

* Approve
* Approve with comments
* Request changes
* Block

---

# Принцип Review

Review оценивает соответствие реализации:

* ADR
* структуре проекта
* data-flow
* правилам проекта

Review не должен дублировать архитектурные решения, уже описанные в документации.
