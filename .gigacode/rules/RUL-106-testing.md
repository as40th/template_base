# Test Strategy

Приоритет типов тестов:
1. Happy path
2. Edge cases
3. Failure cases
4. Concurrency / async behavior (если применимо)

---

# Principles

## Isolation

Тесты не должны зависеть от:
- внешних API
- сети
- оперативной системы
- реальной базы данных (если не integration test)

Использовать:
- mocks
- test containers (если требуется)

---

## Determinism

Тесты должны быть:
- полностью детерминированными
- независимыми друг от друга

---

## Structure

Каждый тест должен:
- иметь ясное имя
- проверять одну поведенческую единицу
- содержать Arrange → Act → Assert

Разрешено использовать:
- параметризованные тесты

---

# Mocks Policy

Разрешено:
- mock внешних сервисов
- mock MCP / LLM / API calls

Запрещено:
- mock бизнес-логики
- mock внутренних функций тестируемого модуля

---s

# Coverage Goals

Приоритет покрытия:
- критическая бизнес-логика
- обработка ошибок
- интеграционные точки

---

# Output Format

Тесты должны быть:
- готовы к запуску без доработки
- сгруппированы по модулям
- размещены в соответствии со структурой проекта
