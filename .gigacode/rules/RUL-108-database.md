# DATABASE ACCESS BEST PRACTICES

## 1. Назначение Database Layer

Database Layer используется для:

* хранения и извлечения данных
* управления транзакциями
* обеспечения целостности данных
* выполнения запросов

Database Layer НЕ используется для:

* бизнес-логики
* orchestration
* принятия бизнес-решений
* интеграции UI и API

---

## 2. Repository = Data Access Only

Repository отвечает только за:

* CRUD операции
* поиск данных
* сохранение данных
* работу с транзакциями

Repository НЕ должен содержать:

* бизнес-правила
* workflow
* orchestration
* вызовы внешних сервисов

---

## 3. Service = Business Logic

Service отвечает за:

* бизнес-правила
* валидацию доменных ограничений
* координацию нескольких repositories
* управление бизнес-транзакциями

Service НЕ должен содержать:

* SQL
* ORM-запросы
* инфраструктурные детали хранения

---

## 4. Transaction Boundaries

Транзакция должна:

* иметь явные границы
* охватывать единый бизнес-сценарий
* завершаться commit или rollback

Запрещено:

* скрытые commit внутри repository
* частичные commit в рамках одной операции
* управление транзакцией из нескольких слоёв одновременно

---

## 5. Query Design

Разрешено:

* целевые запросы под конкретный use-case
* выбор только необходимых полей
* пагинация больших выборок

Запрещено:

* SELECT *
* загрузка лишних данных
* N+1 queries
* неоптимизированные циклические запросы

---

## 6. Domain Model Separation

Доменные модели должны быть отделены от моделей хранения.

Разрешено:

* mapping ORM → Domain
* mapping Domain → ORM

Запрещено:

* распространение ORM сущностей по всему приложению
* связывание бизнес-логики с ORM API

---

## 7. Error Handling

Слой данных должен явно обрабатывать:

* not found
* unique constraint violations
* optimistic locking conflicts
* connection failures

Запрещено:

* silent failures
* подавление database exceptions
* возврат null вместо контролируемой ошибки

---

## 8. Performance

Каждый запрос должен:

* иметь понятную цель
* использовать индексы при необходимости
* масштабироваться с ростом данных

Необходимо избегать:

* полного сканирования больших таблиц
* избыточных join
* повторных одинаковых запросов

---

## 9. Observability

Каждая операция с БД должна быть наблюдаемой.

Логируются:

* trace_id
* длительность запроса
* количество затронутых записей
* ошибки выполнения

Запрещено:

* скрытое выполнение запросов
* потеря контекста запроса

---

## 10. Migration Rules

Миграции используются только для:

* изменения схемы
* миграции данных
* создания индексов и ограничений

Каждая миграция должна:

* быть идемпотентной при повторном запуске где возможно
* иметь rollback стратегию
* выполняться независимо

Запрещено:

* бизнес-логика внутри миграций
* зависимости от внешних сервисов
* ручные действия без документации

---

## 11. Definition of Done

* repository содержит только data access
* service содержит бизнес-логику
* транзакции имеют явные границы
* ORM не протекает в бизнес-слой
* запросы оптимизированы
* ошибки обрабатываются явно
* операции наблюдаемы
* миграции воспроизводимы
* отсутствуют скрытые side effects
